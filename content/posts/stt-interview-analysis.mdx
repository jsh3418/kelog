이 글은 기존 제품 파이프라인의 구조적 문제를 발견하고, 기획 단계에서 해결책을 제안하여 고객 온보딩을 개선한 경험을 정리한 글입니다.

## 배경

피드인 프로젝트의 핵심 가치는 인터뷰 데이터를 분석하고 액션 아이템을 도출하는 것입니다.
기존 플로우는 챗봇을 통해 사용자 데이터를 수집한 뒤, 이를 분석 파이프라인에 넘기는 구조였습니다.

```text
[챗봇 인터뷰] → [텍스트 데이터 수집] → [분석 파이프라인] → [액션 아이템 도출]
```

## 문제: 서비스 효용을 체감하기까지의 긴 여정

신규 고객사 입장에서 이 플로우에는 구조적인 문제가 있었습니다.

챗봇을 통한 데이터 수집이 완료되어야만 분석 결과를 확인할 수 있기 때문에, 서비스의 핵심 가치를 체감하기까지 오랜 시간이 걸릴 것이 예상되었습니다. 고객사가 "이 서비스가 우리에게 어떤 가치를 주는가?"를 판단하려면 수일에서 수주간의 데이터 수집 과정을 먼저 거쳐야 했습니다.

```
[신규 고객 온보딩]
    ↓
[챗봇 배포 및 데이터 수집 시작]  ← 수일~수주 소요
    ↓
[충분한 데이터 축적]
    ↓
[분석 결과 확인]  ← 이 시점에서야 서비스 가치 체감
```

즉, **서비스의 가치를 증명하기 위해 고객에게 먼저 시간 투자를 요구**하는 구조였습니다.

## 해결: 기존 녹음 파일을 활용한 즉시 분석

핵심 아이디어는 간단합니다.
"고객사는 이미 과거 인터뷰 녹음 파일을 보유하고 있으므로, 이를 텍스트로 변환해서 기존 분석 파이프라인에 바로 연결하면 된다."

```
[기존 녹음 파일 업로드] → [STT 변환] → [분석 파이프라인] → [액션 아이템 도출]
```

데이터 수집 과정을 건너뛰고, 고객이 이미 가진 자산을 활용해 서비스 가치를 즉시 보여줄 수 있습니다.

## 개선 효과

기존 플로우와 비교하면 고객 경험이 근본적으로 달라졌습니다.

```
[기존] 온보딩 → 챗봇 배포 → 수일~수주 데이터 수집 → 분석 결과 확인
[개선] 온보딩 → 녹음 파일 업로드 → 즉시 분석 결과 확인
```

- **즉각적인 가치 체감**: 신규 고객이 데이터 수집 과정 없이도 서비스의 핵심 분석 기능을 바로 경험할 수 있게 되었습니다.
- **기존 자산 활용**: 고객이 이미 보유한 녹음 파일을 활용하므로, 추가적인 시간 투자 없이 서비스 도입을 검토할 수 있습니다.

## 회고

이 기능은 기술적 난이도보다 **문제 인식** 자체가 핵심이었습니다.

분석 기능을 테스트하던 중, 우리가 이미 보유한 실제 인터뷰 데이터를 그대로 넣으면 유의미한 분석 결과가 나오겠다는 아이디어에서 출발했습니다. 녹음 파일 형태였기 때문에 자연스럽게 STT API를 떠올렸고, 이를 회의에서 제안하여 고객에게 가치를 빠르게 전달할 수 있는 기능 기획으로 이어졌습니다.

Whisper API 연동 자체는 어렵지 않았습니다. 어려운 건 "고객이 왜 이탈하는가"를 파악하고, 그 원인에 맞는 해결책을 기획하는 과정이었습니다.

다만 아쉬운 점은 STT API의 비용입니다. 녹음 파일의 길이와 건수에 따라 비용이 선형적으로 증가하기 때문에, 고객사가 대량의 인터뷰 데이터를 보유한 경우 비용 부담이 커질 수 있습니다. 향후 자체 STT 모델 도입이나 비용 효율적인 대안을 검토할 필요가 있다고 느꼈습니다.
